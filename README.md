一、文件與 AI 回覆語言規範（Documentation & AI Response Rules，文件與 AI 回覆規範）

為確保溝通一致性與可讀性，專案採用以下規範。

1. 英文名詞標註規則

在所有 非程式碼 的文件與 AI 回覆中：

所有 英文專有名詞或技術名詞

必須在後方以中文括號標註說明

格式範例
LocalStorage（瀏覽器本地儲存）
Idle Loop（放置型核心循環）
State（全局狀態）
Skin（主題／外觀配置）

2. 適用範圍

本規範適用於：

AI 回覆內容

技術設計文件

規則與系統說明文件

3. 例外項目

以下情況可不標註中文：

程式碼區塊
變數名稱
函式名稱
檔名與路徑

二、AI 生成程式碼註解與教學規範
（AI Code Annotation & Learning Rules，AI 程式碼註解與學習規範）

規範目的

本專案開發者正在自學程式設計，為確保理解與學習效果，所有由 AI 生成的程式碼必須同時具備「可用性」與「可讀性」，不得僅提供可執行結果而缺乏說明。

1. 程式碼逐條說明要求

在 AI 回覆中，凡提供 完整程式碼或程式碼片段，必須：
對 每一行或每一個邏輯區塊
逐條說明其「用途」與「影響範圍」
說明內容需回答至少一個問題：
這一行是做什麼用的？
它控制或影響了哪個行為、數值或狀態？

2. 必須說明的重點項目

AI 在解釋程式碼時，必須明確指出以下內容（如適用）：
變數的用途（Variable，用途）
函式的責任（Function Responsibility，函式職責）
條件判斷控制的情況（Condition，條件）
狀態改變的影響（State Change，狀態變化）
與遊戲系統的關聯（例如資源、技能、存檔）

3. 禁止行為

以下情況視為 不符合本專案規範：
僅貼出程式碼，沒有逐條解釋
使用大量專業術語但未解釋其意義
假設讀者「本來就應該懂」
將「這是標準寫法」作為唯一說明

4. 教學導向原則（Learning-first Principle，學習優先原則）

說明應以「初學者可理解」為優先
寧可多解釋，也不要省略關鍵邏輯
若存在多種寫法，需說明為何選擇目前這一種

5. 與其他規範的關係

本規範 優先於簡潔性
本規範 不影響程式碼效能要求

本規範僅適用於：

AI 生成內容
教學與設計討論階段

【AI 合約】

任何 AI 回覆，若未同時符合：
- CONTEXT 規則
- README 規範
- 教學導向說明

視為無效回覆，需重新生成。
